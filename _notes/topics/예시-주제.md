## ❓ 시작 질문
단 3명이, 문서도 없는 상태에서, 4개국에 서비스를 런칭하고 운영까지 하는 게 가능할까?

## 🧩 조각 1 — 왜 이게 불가능해 보였나?
- 유럽 시작으로 글로벌 커넥티드카 서비스 런칭 필요
- 기존 한국·미국·중국이 완전 분리 개발, 서버쪽 공식 문서 거의 없음
- 오토에버 통해 2차 협력사가 개발 → 인도·중국 등 파편화, 일원화 난제
- 비즈니스로 보면: 나라마다 신규 런칭/업데이트에 막대한 비용 (앱까지 10억+)
- 핵심 문장: 파편화는 기술 문제가 아니라 매번 돈이 새는 비즈니스 문제였다

## 🧩 조각 2 — 3명이 어떻게 나눴나?
- 목표: 보안성 높고 플랫폼 기반 일원화 → 원앱 글로벌 런칭
- 서버 언어: 성능 위주는 Go, 데이터 처리 많은 것은 Node.js
- 보안: 계정 OAuth + 파트너사별 JWT 인증·인가, DMZ존 API Gateway로 tyk
- 오픈소스만 활용, API 추가 시 CI/CD에서 tyk 자동 반영
- 구성: API G/W · Account · Vehicle · Partner · App 서버 — 단 3명

## 🧩 조각 3 — 백업 플랜이 없다는 것
- 실패하면 차량 출시에 문제. 백업 플랜 없음
- 해외 해킹 시도·타사 계정 데이터 유출 뉴스마다 심장 멎는 스트레스
- 하지만 이게 나를 단단하게 만든 경험
- 핵심 문장: 물러설 곳이 없을 때 사람은 가장 빠르게 배운다

## 🧩 마지막 조각 — 완성된 그림
- 유럽→한국→중국→인도까지 2018년까지 차례로 런칭 + 직접 운영
- 소수정예가 가능했던 건 "표준·오픈소스 위에 올리고, 자동화로 운영 부담을 줄인" 덕분
- 답: 가능하다. 단 모든 것을 직접 만들지 않고, 검증된 것 위에 최소한만 새로 지을 때

## 🎁 가져갈 것
1. 파편화 비용은 기술이 아니라 돈으로 환산해 설득하라
2. 소수정예의 무기는 표준·오픈소스·자동화 — 새로 만들 양을 줄이는 것
3. 백업 플랜이 없는 프로젝트일수록 관측·보안 자동화에 먼저 투자하라

## 💡 설명이 필요한 개념
- JWT: 파트너사마다 다른 권한을 주는 서명된 출입증
- API Gateway (tyk): 모든 요청이 통과하는 정문, DMZ존에서 실행
- DMZ: 외부와 내부 사이 완충 지대
