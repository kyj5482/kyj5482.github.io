# 03. 콘텐츠 시스템 — AI 에이전트 생산 파이프라인

> 핵심 철학: **내가 하는 일은 "경험 입력"과 "검수" 두 가지뿐이다.** 나머지는 전부 에이전트가 한다.
> 내 시간 예산: 주 2~3시간 (입력 30분 + 검수 60분 + 발행/소통 30~60분)

## 1. 생산 파이프라인

```
[1] 경험 캡처 (본인, 수시)
    음성메모/휴대폰 메모 — 형식 무시, 생각나는 대로
        ↓
[2] 초안 생성 (에이전트)
    _branding 문서 컨텍스트 + 캡처 메모 → 블로그 글 초안 (.md, front matter 포함)
        ↓
[3] 검수 (본인, 15–30분)
    사실관계 / 회사 보안 / 가족 프라이버시 / 내 목소리인가
        ↓
[4] 발행 (에이전트)
    _posts 커밋·푸시 → GitHub Pages 자동 배포
        ↓
[5] 리퍼포징 (에이전트) — 원본 1개에서 파생
    ├─ LinkedIn 포스트 (요약 + 훅, 한/영)
    ├─ 뉴스레터 섹션 (다이제스트용)
    └─ 쇼츠 스크립트 (B트랙 위주: 훅 3초 + 본문 30초 + CTA)
```

**1 원본 → 4 산출물.** 채널별로 콘텐츠를 새로 만드는 순간 루틴은 무너진다.

### 허브와 퍼널 규칙 (모든 산출물에 적용)

1. **GitHub이 허브**: 원본 글과 코드는 항상 GitHub에 남는다. 외부 채널 산출물에는 반드시
   "더 깊이 보려면" 링크(블로그 글 또는 swim.* 레포)를 넣어 GitHub으로 되돌아오게 한다.
2. **뉴스레터가 전환점**: 모든 산출물의 마지막 CTA는 뉴스레터 구독 링크다.
   이메일 리스트는 알고리즘에 좌우되지 않는 유일한 소유 자산이다.
3. **레벨 규칙** (01 문서 4-1): 초기 발행 비중 L1 50% / L2 30% / L3 20%.
   새 시리즈는 L1로 시작하고, L1 글 하단에 같은 시리즈의 다음 레벨 링크를 건다.

## 2. 주간 루틴

| 요일 | 할 일 | 시간 | 누가 |
|------|------|------|------|
| 토/일 | 이번 주 주제 선택(백로그에서) + 음성메모 입력 | 30분 | 본인 |
| 일 | 블로그 초안 + 리퍼포징 3종 생성 | — | 에이전트 |
| 월 | 초안 검수·수정 지시 → 발행 | 30분 | 본인 |
| 화 | LinkedIn 게시 | 5분 | 본인 (게시 버튼만) |
| 목 | 쇼츠 게시 (Phase 1부터) | 10분 | 본인 |
| 격주 금 | 뉴스레터 발송 (Phase 1부터) | 20분 검수 | 본인 |

주말에 대회 원정이 있는 주는 **대회 자체가 콘텐츠 소스다** — 이동 중 음성메모 3개면 그 주 콘텐츠가 나온다.

## 3. 에이전트 프롬프트 템플릿

### 3-1. 블로그 글 초안

```
_branding/01-brand-strategy.md 와 03-content-system.md 를 읽어라.
아래 내 메모를 바탕으로 [A트랙/B트랙] 블로그 글 초안을 _posts 에 작성해줘.

- 주제: (04-topic-backlog의 번호 또는 새 주제)
- 내 메모: (음성메모 텍스트 붙여넣기)
- 분량: 1,500~2,500자, 소제목 3~5개
- 톤: 경험자의 담백한 1인칭. 과장·클리셰 금지. "~것 같다" 남발 금지.
- front matter: Chirpy 형식 (categories, tags, image 자리 표시)
- 마지막에 발행 전 체크 3항목(사실관계/보안/프라이버시)을 코멘트로 붙여줘.
```

### 3-2. LinkedIn 리퍼포징

```
방금 발행한 _posts/(파일명) 글을 LinkedIn 포스트로 재가공해줘.
- 첫 줄은 스크롤을 멈추게 하는 훅 (질문형 또는 반전형)
- 본문 8~12줄, 줄바꿈 자주, 이모지 최소
- 한국어 버전 + 영어 버전
- 마지막 줄: 블로그 링크 CTA
```

### 3-3. 쇼츠 스크립트 (B트랙)

```
(주제/메모) 로 60초 쇼츠 스크립트를 작성해줘.
- 구조: 훅(3초, 궁금증 유발) → 핵심 1가지(40초) → CTA(다음 편 예고 or 앱)
- 자막용 문장 단위로 분리, 화면 지시 포함 (예: [자료화면: 대회장])
- 아이 얼굴 노출 없이 만들 수 있는 구성일 것
```

### 3-4. 뉴스레터 (격주)

```
지난 2주간 _posts 발행 글과 내가 준 근황 메모로 뉴스레터 초안을 만들어줘.
구성: ① 오프닝 한 단락(근황) ② A트랙 글 요약+링크 ③ B트랙 코너 ④ Build in Public 진행 상황 ⑤ 마무리 질문(회신 유도)
```

## 4. 콘텐츠 카테고리 구조 (블로그)

| 카테고리 | 트랙 | 내용 |
|---------|------|------|
| `[Developing]` | A | 기술 경험담, 아키텍처, AI 에이전트 활용 |
| `[Building]` | A×B | Build in Public — 수영 앱 개발 일지 |
| `[Swim Parent]` | B | 수영 선수 부모 경험, 한·미 비교, 대회 원정기 |
| `[Startup]` | A | 1인 스타트업 전략, 회고, 분기 리뷰 |

## 5. 발행 전 최종 체크 (본인이 직접, 매번)

- [ ] 사실관계: 숫자·연도·회사명이 정확한가, 공개된 정보 수준인가
- [ ] 회사 보안: 재직사 내부 수치/미출시 계획이 없는가 (01 문서 가드레일 6-1)
- [ ] 아이들 자산성: 이 콘텐츠가 훗날 아이들 본인의 브랜딩에 자산이 되는가 — 성장 서사인가, 지우고 싶을 서술은 없는가 (가드레일 6-2)
- [ ] 퍼널: GitHub 링크와 뉴스레터 CTA가 들어 있는가
- [ ] 내 목소리: 소리 내 읽었을 때 "내가 쓴 글"로 들리는가
