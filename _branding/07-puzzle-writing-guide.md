# 07. 퍼즐 글쓰기 가이드 — 직접 쓰는 법

> 목표: **본인이 직접 써도, 에이전트가 써도, 항상 같은 퍼즐 구조가 나오도록** 규칙과 템플릿을 고정한다.
> 이 파일 하나만 보면 조각 준비 → MD 작성 → 발행까지 끝난다. (지속가능성의 핵심은 "매번 고민하지 않는 것")

---

## 1. 퍼즐 글의 뼈대 (항상 동일)

```
질문(❓)  →  조각1 🧩  →  조각2 🧩  →  조각3 🧩  →  완성된 그림 🧩  →  가져갈 것 🎁
 (궁금증)    (부분답)     (부분답)     (부분답)       (전체답)          (독자 행동)
```

- **하나의 질문**만 던진다. 여러 개면 퍼즐이 흩어진다.
- **조각은 3~4개.** 각 조각은 질문의 "부분 답"이고, 조각마다 **핵심 한 문장**(굵게)을 박는다.
- 마지막 조각에서 부분 답들이 합쳐져 **질문의 전체 답**이 된다.
- 🎁 가져갈 것은 감상이 아니라 **독자가 오늘 할 수 있는 행동** 3개.

---

## 2. 글쓰기 전, 조각 준비하기 (5칸 표 채우기)

글을 쓰기 전에 **아래 5칸만 메모**하면 된다. 이게 조각 준비의 전부다.
문장이 아니어도 되고, 음성메모로 불러도 된다.

| 칸 | 채울 내용 | 예시 (계정 통합 글) |
|----|----------|---------------------|
| ❓ 질문 | 독자가 궁금해할 단 하나 | "해본 적 없는 일을 혼자 6개월에 해내려면?" |
| 🧩 조각1 | 상황/문제의 진짜 정체 | "브랜드 3개 = 회사 3개, 기술이 아니라 조율 문제" |
| 🧩 조각2 | 어떻게 접근했나 | "모르는 채로 시작 — 배우는 순서를 설계" |
| 🧩 조각3 | 대가/전환점/교훈 | "새벽 3시의 대가, 반복 불가능한 성공" |
| 🎁 가져갈 것 | 독자 행동 3개 | "구조부터 읽어라 / 배우며 전진 / 대가를 계산하라" |

> 조각이 4개 필요한 글(예: EP.1)이면 조각3을 둘로 나눈다. 규칙은 "질문 1개 + 조각 3~4개 + 완성 + 가져갈 것".

### 조각을 잘 나누는 3가지 기준
1. **시간순**이 아니라 **궁금증 순**으로 나눈다 — "그래서 다음은?"이 나오는 지점이 조각 경계다.
2. 각 조각은 **하나의 소질문**으로 제목을 단다 (`## 🧩 조각 N — 왜 …?`). 소질문이 안 나오면 그 조각은 쪼개거나 합친다.
3. 조각 끝마다 **한 문장으로 요약**되면 성공. 안 되면 내용이 두 개 섞인 것.

## 3. 개념 설명 & 레퍼런스 (논문인데 이해되는 흐름)

전문 용어가 나오면 흐름을 끊지 말고 **개념 카드**로 옆에 붙인다. 독자는 몰라도 읽히고, 궁금하면 펼친다.

```markdown
> 💡 **개념 — OAuth 2.0**
> 비밀번호를 넘기지 않고 "이 앱에 내 정보 접근을 허용"만 위임하는 표준.
> 호텔 방 키를 복제해 주는 대신, 정해진 층만 열리는 임시 카드를 주는 것과 같다.
> 📖 더 보기: [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749)
```

레퍼런스 원칙:
- 개념마다 **한 줄 비유 + 한 줄 정의 + 신뢰할 링크 1개**. 링크는 공식 표준(RFC/W3C), 원 논문, 공식 문서, 위키피디아 순으로 신뢰도 높은 것.
- 링크는 **실제 존재하는 URL만** (RFC, 공식 docs, 위키). 지어내지 않는다 — 발행 전 체크 1번(사실관계)에 포함.
- 글 맨 아래 `## 📚 더 깊이 읽을거리`에 3~5개를 모아 논문식 참고문헌처럼 정리한다.

자주 쓰는 개념 카드는 8절에 사전으로 모아두었다 — 복붙해서 쓴다.

## 4. 분량 가이드 (짧아서 인사이트가 안 남는 문제 해결)

- **전체 2,500~4,000자** (기존 글은 1,500자대라 얇았다. 최소 2,500자로 올린다.)
- 조각당 **400~700자**: ① 장면/사실(내 경험) → ② 왜 그랬나(맥락·개념) → ③ 그래서 뭐(핵심 문장)
- "무슨 일이 있었다"로 끝내지 말고 **"그래서 당신에게 무엇을 의미하나"**까지 간다. 이게 인사이트의 정체다.
- 숫자·고유명사를 아끼지 말 것 (100여 대 서버, 연 2M 달러, 3명, 6개월). 구체성이 신뢰다.

### MD 포맷 체크 (자주 깨지는 곳)
- front matter는 **`---` 사이에** 정확히. `date`는 `YYYY-MM-DD HH:MM:SS +0900` 형식.
- **`image:` 필드는 넣지 않는다** — 퍼즐 히어로가 대표 이미지를 대신한다. (넣으면 이미지가 두 번 나온다)
- `puzzle-hero` div와 `<script>` 두 줄은 **본문 맨 위**에 그대로.
- 개념 카드는 `> 💡` 인용문, 가져갈 것은 `<div class="takeaway">` HTML — 이 두 개는 형식을 지켜야 스타일이 입혀진다.
- `##` 제목 앞뒤로 빈 줄 1개씩. 목록(`-`, `1.`) 앞에도 빈 줄.

## 6. 히어로 이미지 준비

- 파일: `assets/img/posts/슬러그.svg` (또는 .png/.jpg). 16:9 비율.
- **퍼즐은 자동**이다 — 이미지만 넣으면 `puzzle.js`가 3×3으로 쪼개 스크롤에 맞춰 공개한다. 이미지 자체는 평범한 한 장이면 된다.
- 제작 방법 3가지:
  1. **에이전트에게 요청** (기본): "이 글 주제로 다크 네이비+시안 톤 히어로 SVG 만들어줘" → 사이트 톤 유지
  2. **생성형 이미지**: Higgsfield 크레딧이 있으면 `generate_image`로. 없으면 1번.
  3. **직접**: 사진/도표를 16:9로 넣어도 됨. 밝은 이미지면 조각 대비가 더 잘 보인다.

## 7. 발행 절차 (지속가능 루틴)

1. 5칸 표 채우기 (본인, 10분) — 또는 음성메모로 에이전트에게
2. 에이전트가 템플릿으로 초안 + 히어로 이미지 생성
3. **검수 (본인, 15분)**: ① 사실관계(00-history-source와 대조, 링크 실재 확인) ② 회사 보안(가드레일) ③ 내 목소리
4. 로컬/PR에서 퍼즐 동작 확인 → main 머지 → 자동 배포
5. `04-topic-backlog.md`의 발행 캘린더 상태 갱신

> 에이전트 호출 예: "_branding/00, 01, 07 을 읽고, 07의 템플릿으로 [주제] 글을 퍼즐 형식으로 써줘.
> 조각 준비 표: (5칸 붙여넣기). 개념 나오면 개념 카드로 설명하고 실재하는 레퍼런스를 붙여줘."

## 8. 개념 카드 사전 (복붙용 — 내 글에 자주 나오는 것들)

> 💡 **개념 — OAuth 2.0**
> 비밀번호를 넘기지 않고 "이 앱에 내 정보 접근을 허용"만 위임하는 인증 표준.
> 📖 [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749)

> 💡 **개념 — JWT (JSON Web Token)**
> 위조 못 하도록 서명된 "출입증". 서버가 매번 DB를 안 봐도 이 증표만으로 신원·권한을 확인한다.
> 📖 [RFC 7519](https://datatracker.ietf.org/doc/html/rfc7519) · [jwt.io](https://jwt.io/introduction)

> 💡 **개념 — API Gateway**
> 모든 요청이 통과하는 정문. 인증·라우팅·유량 제어를 한곳에서 처리해 뒤쪽 서비스를 보호한다.
> 📖 [tyk.io 문서](https://tyk.io/docs/) · [Microservices.io — API Gateway](https://microservices.io/patterns/apigateway.html)

> 💡 **개념 — RTOS (실시간 운영체제)**
> "정해진 시간 안에 반드시" 반응해야 하는 기기(자동차·가전)를 위한 OS. 빠름보다 **예측 가능성**이 핵심.
> 📖 [위키피디아 — RTOS](https://en.wikipedia.org/wiki/Real-time_operating_system)

> 💡 **개념 — 디지털 트윈**
> 현실의 사물(자동차)을 소프트웨어로 실시간 복제한 쌍둥이. 실물을 안 뜯어도 상태를 진단·예측한다.
> 📖 [위키피디아 — Digital twin](https://en.wikipedia.org/wiki/Digital_twin)

> 💡 **개념 — 컴퓨테이션 오프로딩**
> 힘 부족한 기기(차량 제어기) 대신 서버가 무거운 계산을 대신 해주는 것. 단말의 성능·메모리 한계를 우회한다.
> 📖 [위키피디아 — Computation offloading](https://en.wikipedia.org/wiki/Computation_offloading)

> 💡 **개념 — 백분위수 (P10/P50/P90)**
> 데이터를 줄 세웠을 때 아래에서 10%/50%(중앙값)/90% 지점의 값. 평균이 숨기는 "분포"와 "이상치"를 드러낸다.
> 📖 [위키피디아 — Percentile](https://en.wikipedia.org/wiki/Percentile)

> 💡 **개념 — 비전 + LLM (멀티모달 추출)**
> 이미지(결과지)를 읽어 의미 있는 데이터(기록)로 바꾸는 AI. 사람이 눈으로 하던 판독을 자동화한다.
> 📖 [위키피디아 — Multimodal learning](https://en.wikipedia.org/wiki/Multimodal_learning)

> 💡 **개념 — 데이터 마트**
> 전사 창고(데이터 웨어하우스)에서 목적별로 잘라낸 작은 매대. 그 서비스가 쓰기 딱 좋게 미리 정리해 둔 것.
> 📖 [위키피디아 — Data mart](https://en.wikipedia.org/wiki/Data_mart)

> 💡 **개념 — 암전류 (자동차 12V)**
> 시동을 꺼도 흐르는 미세 전류. 과하면 배터리가 방전된다. 12V 전압을 지속 모니터링하면 방전 징후를 미리 잡는다.
> 📖 [위키피디아 — Parasitic draw](https://en.wikipedia.org/wiki/Parasitic_battery_drain)
